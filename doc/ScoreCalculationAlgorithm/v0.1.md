---                                                                                                                                                                                                  
  TL;DR                                                                                                                                                                                                
                                                                                                                                                                                                       
  用 Weighted RRF + Chorus 显式加成 + Min-Max 二次归一化到 0-1，暴露 score ∈ [0,1] 给 LLM；不用 native score 做主排序，只把它作为可选 tiebreaker。                                                     
                                                                                                                                                                                                     
  公式（k=60，下面给完整推导）：                                                                                                                                                                       

  raw(d) = Σ_p  w_p · 1/(60 + rank_p(d)) · chorus_boost(|H(d)|)
  score(d) = (raw(d) - raw_min) / (raw_max - raw_min)    // 输出层归一化

  理由用一句话说清：LLM 在 RAG 里对绝对分数不敏感、对顺序敏感（Cuconasu et al. SIGIR'24 The Power of Noise 实证），所以融合算法只需要保证"相对秩可信 + 量纲对人友好"，不需要标定绝对概率。RRF
  是这个目标的最低成本最优解，weighted 修正 provider 质量差异，chorus boost 显式编码"多家共识"，最后 min-max 把那个让你不爽的 0.04 数字变成 0-1 让 LLM 容易消费。

  为什么不是 CombMNZ：你有 Brave 和 Gemini 两家不返回 score，强行用 CombSUM/CombMNZ 就要给它们伪造分数（例如从 rank 反推 1 - rank/N），这相当于在 score 空间里做了一次粗糙的 RRF，不如直接 RRF 干净。

  为什么不是纯 Linear Combination：Vogt & Cottrell 1999 给出 LC 优于单系统的三个充要条件：(1) 至少一家高质量、(2) 相关文档高重叠、(3) 不相关文档低重叠。条件 2、3 在你这个 7 家异质 SERP
  场景下基本不满足（Tavily LLM-optimized vs Brave 通用 vs Exa neural 的相关项分布差距巨大）。

  ---
  推荐公式

  const K = 60;                    // RRF 常量，Cormack '09 经验值
  const ALPHA = 0.4;               // Chorus boost 强度

  // provider 质量权重（基于离线评测或 default）
  const W: Record<Provider, number> = {
    tavily:   1.0,   // LLM-optimized SERP, 高基线
    exa:      0.9,   // neural, 长 query 强、短 query 弱
    brave:    0.7,   // 通用 SERP, 噪声较多
    jina:     0.7,
    searxng:  0.6,   // 已经是元搜索，二次融合降权
    firecrawl:0.7,
    gemini:   1.0,   // grounding 高质量但样本量小
  };

  // 单条 url 的融合分
  function fuse(urlHits: Hit[]): number {
    const sources = urlHits.map(h => h.provider);
    const n = new Set(sources).size;             // |H(d)|

    // 1. Weighted RRF 主项
    const rrf = urlHits.reduce(
      (s, h) => s + W[h.provider] / (K + h.rank),
      0
    );

    // 2. Chorus boost: sqrt(n) 平滑递增，避免线性放大噪声
    const chorus = Math.pow(n, ALPHA);            // n=1 → 1.0; n=3 → 1.55; n=7 → 2.18

    return rrf * chorus;
  }

  // 全局二次归一化（暴露给 LLM 用）
  function normalize(scores: number[]): number[] {
    const max = Math.max(...scores);
    const min = Math.min(...scores);
    if (max === min) return scores.map(() => 1.0);
    return scores.map(s => (s - min) / (max - min));
  }

  关键常数取值：
  - K = 60：原论文经验值，ES/LangChain/Vespa/Qdrant 全部沿用。你的 top-10 短列表可以下调到 30-40（让 rank 1 vs rank 5 的差距更显著），但建议先用 60 跑 baseline。
  - ALPHA = 0.4：sqrt(n) ≈ n^0.5 太陡（被 7 家命中的 url 比被 1 家命中的高 2.6 倍），log(1+n) 又太弱。0.4 在我看到的 SearXNG 等启发式里是合理中间值。
  - W 权重：上面是基于公开声誉的 default，真正生产用必须用离线评测集（30-50 个标注 query × NDCG@10）调优，否则只是凭感觉。

  边界条件：
  - 只有 1 家命中：chorus = 1（n^0.4 当 n=1 时为 1），分数完全由 weighted RRF 决定
  - 缺失 native score：本方案根本不依赖 native score，自动兼容
  - N 家命中：chorus = N^0.4，对 7 家上限是 2.18 倍，不会溢出

  7 个子问题逐个回答

  1. 算法选型

  如上表。Weighted RRF 是 SOTA 起点，全行业开源基础设施统一选择（Elasticsearch 8.16+ 内置 weight 字段、LangChain EnsembleRetriever 默认 c=60、Qdrant rrf 默认 k=60、Vespa reciprocal_rank 默认
  k=60、LlamaIndex QueryFusionRetriever 同默认）。

  2. Native score 归一化

  直接放弃 native score 作为主排序信号。三个理由：

  1. Brave + Gemini 两家不返回 score，伪造任何一种归一化都引入偏差
  2. Tavily 0-1 是私有评分，Exa neural score 也是私有评分，分布本就不可比；Jain et al. 2005 和 Scheirer et al. 2010 ECCV 都证明跨系统 score normalization 是个公认难题，min-max/z-score 在 outlier
  下都崩
  3. RAG 场景下 LLM 用的是顺序信号，不是分数信号

  替代方案：把 native score 仅作为同 provider 内部 rank 的 tiebreaker（其实 provider 自己已经按 score 排好了，所以这一步通常无操作）。输出层用 min-max 把最终的 weighted RRF 分归一化到 [0,1]，让 LLM
  看到的是 0.0-1.0 而不是 0.04。

  3. Provider 权重设计

  默认拍权重 + 离线评测调优，业界没有公开 default。可参考的硬数据：

  - ES 8.16 / Qdrant / LangChain 都允许 weight 但官方文档不给推荐值，明确说"在你的 eval set 上调"
  - SearXNG 的 settings.yml 默认所有引擎 weight=1（等权）
  - Weaviate 的 alpha 默认 0.5-0.75（向量 vs BM25 倾向向量）

  实操路径：
  - 冷启动：按"LLM-optimized > 自建 neural > 通用 SERP > 二次元搜索"梯度，给 1.0 / 0.9 / 0.7 / 0.6
  - 有 30-50 标注 query：grid search w_p ∈ {0.5, 0.7, 1.0, 1.3} 最大化 NDCG@10
  - 有点击日志：用 LambdaMART (Burges 2010) 学权重，是 e-commerce SOTA（Target.com 2025 production 报告 +2.85% conversion）

  4. 多家命中加成（Chorus Effect）

  理论：Vogt & Cottrell 1999 Fusion Via a Linear Combination of Scores 提出三效应：Chorus Effect（合唱：多家共识）、Skimming Effect（撇脂：取每家 top）、Dark Horse Effect（黑马：单家异常准）。CombMNZ
   中的 × |H(d)| 是最早的显式 Chorus 编码。

  形状选择：
  - 线性 × n：n=7 时放大 7 倍，让一个 7 家命中的中等 url 压过单家命中的 top 1，过激
  - log log(1+n)：n=7 → 2.08，n=1 → 0.69，对单家命中惩罚太重
  - sqrt 或 n^0.4：n=7 → 2.65 / 2.18，n=1 → 1.0，渐进且保守，推荐

  Query 类型相关性：理论上事实类 query 该重 Chorus，探索类该重 Dark Horse。但生产系统里做 query intent classification 又是一层不稳定的依赖，建议先用静态 ALPHA=0.4，等收集到点击数据再做条件加成。

  5. AI 综合答案融入

  不要把 Tavily answer / Gemini grounding text 塞进 results[] 排队。业界的一致做法是顶层并列：

  {
    query: "...",
    ai_summary: {                  // 来自 Tavily answer 或 Gemini text
      text: "...",
      sources: [url1, url2, ...]   // 引用的 url 集合
    },
    results: [...]                  // 融合后的 SERP 列表
  }

  参考：Tavily 响应是 {query, answer, results, ...} 同级；Gemini 用 groundingMetadata.groundingSupports.segment.startIndex/endIndex 把文本范围映射到 chunk index；Perplexity Sonar 用内联 [n] 编号 +
  citations[]；OpenAI Responses 用 annotations[] 的 url_citation 对象。

  额外加成：被 ai_summary.sources 引用的 url，在 results[] 里给一个 +1 命中加成（视作"AI 综合源"那一家也命中了它），让被 AI 综合引用的 url 在 SERP 里自然上浮。

  6. score 字段对 LLM 的真实价值

  关键论文：Cuconasu et al. 2024 The Power of Noise: Redefining Retrieval for RAG Systems (SIGIR'24) 证明 LLM 在 RAG 下：
  - 对召回顺序敏感（reorder 同样的 docs，answer quality 显著变化）
  - 对绝对分数不敏感（给/不给 score、给 0.04/给 0.92，模型决策几乎一致）
  - 对位置偏置敏感（lost-in-the-middle 现象）

  结论：暴露 0.0-1.0 normalized score 是最优解：
  - 不像原始 RRF 那样 0.01-0.05 让 LLM 看着像噪声
  - 也不浪费 token（一个浮点比一长串 rank 元数据短）
  - 给 LLM 一个相对可视的"置信度"信号供 prompting（e.g. "只引用 score > 0.5 的来源"）

  别 expose 原始 RRF 累加值，那是个无上界、量纲依赖 provider 数量的内部数。

  7. 最终公式

  见上面 TypeScript 代码块。完整数学表达：

  $$
  \text{raw}(d) = \left( \sum_{p \in H(d)} \frac{w_p}{60 + r_p(d)} \right) \cdot |H(d)|^{0.4}
  $$

  $$
  \text{score}(d) = \frac{\text{raw}(d) - \min_d \text{raw}(d)}{\max_d \text{raw}(d) - \min_d \text{raw}(d)}
  $$

  输出 results[] 字段：

  {
    url, title, snippet, content?,
    score: 0.87,           // normalized 0-1, 给 LLM 看的
    rank: 1,               // 融合后的全局 rank
    sources: ['tavily', 'exa', 'brave'],   // chorus 证据
    // 可选 debug 字段，prod 可隐藏
    _raw_rrf: 0.0489,
    _per_provider: { tavily: 1, exa: 2, brave: 1 }
  }

  进阶方向（可选）

  如果预算允许，在融合后接一个 cross-encoder reranker（Cohere Rerank 3 / bge-reranker-v2-m3 / Jina Reranker v2）做精排，是当前 RAG 端到端 SOTA：

  7 家 provider × top-10 → 去重 → Weighted RRF → 候选池 ~30 → cross-encoder 重排 → top-K → LLM

  BigDataBoutique 2025、LocalAIMaster 2026 production 报告显示 NDCG@10 提升 5-15 个点、p95 < 200ms。这一步比换 fusion 算法本身的提升大。Tavily 自己的 hybrid SDK 默认就是 Cohere rerank。
